# Undo / Redo for Layout Edits

**Date:** 2026-05-16
**Status:** ✅ Complete
**Impact:** High (core editing UX) | **Risk:** Low (additive — no existing edit logic changed)

## Summary

Implemented genuine undo/redo for the track editor. `FEATURES.md` and
`KEYBOARD_SHORTCUTS.md` advertised undo/redo (`Ctrl+Z` / `Ctrl+Y`) but no
implementation existed — the shortcuts did nothing and a misclick-delete was
unrecoverable. This closes that gap.

## Approach

A new non-persisted `useHistoryStore` keeps `past` / `future` stacks of
**combined layout snapshots**. Each snapshot deep-clones (`structuredClone`)
the three persisted layout stores:

- **track** — `nodes`, `edges`
- **logic** — `sensors`, `signals`, `wires`
- **budget** — `balance`, `totalSpent`, `startingBudget`

Logic and budget are included because `removeTrack` cascade-deletes orphaned
sensors/signals and placement spends budget — a snapshot of only the track
graph would desync those on undo.

Each undoable gesture calls `useHistoryStore.getState().record()` **once,
before** mutating state. `undo()` / `redo()` swap the live stores with a
stacked snapshot and rebuild the spatial indices. History is per-session
(stacks are not persisted); capped at 50 steps.

## Scope Notes

- **Covered:** track placement, deletion, connect-mode joins, and
  sensor/signal/wire add/remove.
- **Not covered (intentional):** switch/signal *toggling* is runtime state, not
  a layout edit. New Layout / template load / file import **reset** the undo
  history (the "New Layout" dialog already warns the action is irreversible).
